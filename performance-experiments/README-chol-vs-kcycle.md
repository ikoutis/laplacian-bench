# Approximate Cholesky vs CMG K-cycle benchmark

A focused, pure-Julia comparison of the approximate-Cholesky solvers from
[Laplacians.jl](https://github.com/danspielman/Laplacians.jl) against the
K-cycle solver in
[CombinatorialMultigrid.jl](https://github.com/ikoutis/CombinatorialMultigrid.jl),
across all the benchmark problem families of this repository.

Solver columns:

| column    | solver |
|-----------|--------|
| `ac`      | `approxchol_lap` / `approxchol_sddm`, split=0 merge=0 (the basic version) |
| `ac-s2m2` | `approxchol_lap` / `approxchol_sddm`, split=2 merge=2 |
| `cmg-k`   | `cmg_solve` with `cycle=:kcycle` (flexible CG, K-cycle preconditioning) |
| `cmg-v`   | `cmg_solve` with `cycle=:vcycle` (same outer loop; reduces to PCG with the classic CMG cycle — the reference that shows what the K-cycle buys) |

Everything here is additive: the original paper scripts (`*_ac.jl`,
`*_nopetsc.jl`, `*_all.jl`, `compareSolvers.jl`) are untouched, results go to
a new subdirectory, and the `.jld2` schema is the same one the existing
analysis notebooks read.

## Files

| file | role |
|------|------|
| `../Project.toml`, `../setup.jl` | slim Julia environment (no MATLAB/PETSc/HyPre) |
| `../julia-files/compareSolversCore.jl` | native-solver harness core (extracted from `compareSolvers.jl`, plus seeding) |
| `../julia-files/cmgSolvers.jl` | SolverTest factories for `ac*` and `cmg-*` columns |
| `../julia-files/benchFamilies.jl` | registry of all 13 problem families with smoke/medium/paper sweeps |
| `chol_vs_kcycle.jl` | the benchmark runner (one family, or `all`) |
| `download_data.jl` | prefetch SuiteSparse/IPM matrices into `../matrix-files/` |
| `run_chol_vs_kcycle.sh` | superscript: run everything, or emit Slurm-array manifests |
| `wulver/` | NJIT Wulver Slurm scripts (setup, array + single sbatch, submit) |
| `summarize_chol_vs_kcycle.jl` | aggregate results into Markdown tables + a TSV |

## Quick start (any machine with Julia ≥ 1.10)

```bash
cd laplacian-bench
julia --project=. setup.jl          # once; adds CombinatorialMultigrid from GitHub
cd performance-experiments
./run_chol_vs_kcycle.sh             # smoke scale: every family, tiny sizes, ~minutes
julia --project=.. summarize_chol_vs_kcycle.jl
```

The registered `CombinatorialMultigrid` release (0.2.0) predates the K-cycle,
so `setup.jl` adds the package from GitHub pinned to a known-good revision.
Overrides: `CMG_DEV=/path/to/checkout` uses a local clone (`Pkg.develop`),
`CMG_REV=<rev>` pins a different branch/commit. Once a `Manifest.toml`
generated by `setup.jl` is committed, `Pkg.instantiate()` alone reproduces
the environment.

## The runner

```
julia --project=.. chol_vs_kcycle.jl <family|all>
    [--scale smoke|medium|paper]   instance sweep size            [smoke]
    [--n 1e6[,1e7]]                chimera size list              [per scale]
    [--reps R]                     repetitions per instance       [3]
    [--seed S]                     base seed                      [1]
    [--tol T]                      relative residual tolerance    [1e-8; spe 5e-9]
    [--maxits K]                   max (outer) iterations         [1000]
    [--solvers ac,ac-s2m2,cmg-k,cmg-v]                            [all four]
    [--limit N]                    cap instances per family
    [--max-hours H]                stop before starting work past H hours
    [--out DIR]                    [../performance-analyses/chol-vs-kcycle]
    [--no-warmup]
```

### Families

| family | matrices | source |
|--------|----------|--------|
| `uniform_grid` | 3-D uniform grids (SDDM), nnz 2e6→2e8 | generated (`uniform_grid_sddm`) |
| `aniso` | anisotropic grids, ξ ∈ 1e-3…1e3 | generated (`aniso_grid_sddm`) |
| `wgrid` | weighted grids, w ∈ 1e-3…1e3 | generated (`wgrid_sddm`) |
| `checkered` | checkered grids, blocks 2…128 | generated (`checkered_grid_sddm`) |
| `sachdeva_star` | star-joined complete graphs, k = 100…800 | generated (`star_join`) |
| `suitesparse` | curated SuiteSparse selection | downloaded (cached) |
| `spe` | SPE oil-reservoir grids 0.5m…16m, tol 5e-9 | **manual** `.mm` files (see `Tutorial.md`) |
| `chimeraIPM` | IPM Laplacians `uc.i*.eps*.*` | downloaded (cached) |
| `spielmanIPM` | IPM Laplacians `sk<k>i<i>` | downloaded (cached) |
| `uni_chimera` / `wted_chimera` | (weighted) chimeras, Laplacian path | generated |
| `uni_bndry_chimera` / `wted_bndry_chimera` | chimeras with boundary, SDDM path | generated |

Missing data files are skipped with a warning (the family then simply
contributes fewer rows); run `download_data.jl` first on a machine with
internet access.

### Repetitions and randomness

- Chimera instances are indexed `i = 1000*seed + rep`, so seeds give disjoint
  graph streams and the same `(seed, rep)` is always the same graph.
- The right-hand side is `b = A*randn(n)` normalized (as in the paper
  scripts), with the RNG seeded per `(seed, testName, rep)`.
- Each solver build re-seeds the global RNG per
  `(seed, testName, rep, solvername)` — `approxchol` is internally randomized,
  so this makes every column reproducible independently of which other
  columns run. (CMG's hierarchy construction is deterministic.)
- Reruns with identical arguments and Julia version are bit-identical;
  `Base.hash` is used in seed derivation, so reproducibility is guaranteed
  per Julia version, not across versions.

### Output

One file per (family, scale/size, seed, reps):
`../performance-analyses/chol-vs-kcycle/<family>.<scale|nSIZE>.seed<S>.reps<R>.jld2`,
saved after **every** repetition (a killed job keeps its finished rows).
Same dictionary schema as the original experiments — per solver
`{name}_solve/_build/_tot/_its/_err` plus `nv/ne/testName/names` — with two
additive columns `rep` and `seed`.

**Iteration-count caveat:** `cmg-k_its` counts *outer* flexible-CG
iterations; each hides budget-capped inner FCG work at coarse levels, and one
`cmg-v`/`ac` iteration is one PCG step. Iteration counts are not
work-comparable across columns — compare `_tot`/`_solve` (and `_err`, which
the harness computes uniformly as `norm(A*x-b)/norm(b)`).

A failed solver run (exception) records `Inf` in that row and the run
continues. The curated benchmark matrices are all SDDM or near-SDDM with
non-positive off-diagonals (`Tutorial.md`), so CMG accepts every one of them —
`cmg_solve`'s `validateInput!` only rejects an asymmetric matrix or a positive
off-diagonal, neither of which occurs here. The three "approximately SDDM"
SuiteSparse matrices (`McRae/ecology1`, `McRae/ecology2`, `HB/nos7`) are merely
slightly not diagonally dominant (~1e-16) and are handled normally. The
try/catch is a general safety net for arbitrary user-supplied inputs. Note the
native harness imposes no wall-clock limit per solver — only `--maxits` and the
Slurm job time bound a pathological case.

## The superscript

```bash
./run_chol_vs_kcycle.sh [--scale smoke|medium|paper] [--reps R] [--seed S]
                        [--run | --emit-manifest] [--julia CMD] [--extra "..."]
```

`--run` (default) executes every task sequentially: the 9 fixed families once
each + the 4 chimera families once per size (paper: 1e4, 1e5, 1e6, 1e7 → 25
tasks). `--emit-manifest` instead writes `wulver/manifest.<scale>.small.txt`
and `.large.txt` — one runner invocation per line — for the Slurm array job.
The large tier holds the memory-hungry tasks (1e7 chimeras, nnz≈2e8 grids,
spe16m). Default reps: paper 3, medium 5, smoke 2.

## Wulver (NJIT) walkthrough

```bash
# on a LOGIN node (compute nodes have no internet):
cd laplacian-bench/performance-experiments/wulver
./setup_wulver.sh --scale paper      # Julia (module or juliaup 1.10), Pkg env, data
# edit chol_vs_kcycle_array.sbatch: --account=PI_UCID_HERE -> your PI account
#   (find it: sacctmgr show associations user=$USER format=account%30)
./submit_wulver.sh --scale paper     # emits manifests, sbatches small + large tiers
squeue -u $USER                      # watch; logs in wulver/logs/
julia --project=../.. ../summarize_chol_vs_kcycle.jl   # when done
```

Job shape: 1 task, `--cpus-per-task=4`, `JULIA_NUM_THREADS=OPENBLAS_NUM_THREADS=1`
(all four solvers are single-threaded — pinning keeps timings comparable).
Small tier `--mem=48G --time=24:00:00`; large tier submitted with
`--mem=200G --time=72:00:00` (an nnz≈2e8 SDDM matrix alone is ≈3 GB; solver
fill-in and multigrid hierarchies multiply that severalfold). Use
`--max-hours` (via `--extra`) if you need the runner to stop cleanly before
the Slurm limit; results are saved after every repetition regardless.

If `$HOME` quota is tight, set `JULIA_DEPOT_PATH` (e.g. to
`/project/<PI>/$USER/.julia`) before `setup_wulver.sh`, and export it in the
sbatch files too.

## Summarizing

```bash
julia --project=.. summarize_chol_vs_kcycle.jl            # default results dir
julia --project=.. summarize_chol_vs_kcycle.jl DIR --tsv out.tsv
```

Prints one Markdown table per result file — median total seconds (median
iterations) per solver per instance, with `[Nf]` marking failed repetitions —
and writes a tab-separated `summary.tsv` (tab-separated because testNames
contain commas) for notebooks/pandas. The raw `.jld2` dictionaries remain
fully compatible with the existing `performance-analyses` notebooks.
